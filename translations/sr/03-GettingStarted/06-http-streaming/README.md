# HTTPS стриминг са Протоколом моделског контекста (MCP)

Ово поглавље пружа свеобухватан водич за имплементацију сигурног, скалабилног и стриминга у реалном времену помоћу Протокола моделског контекста (MCP) преко HTTPS. Покрива мотивацију за стриминг, доступне транспортне механизме, како имплементирати стриминг HTTP у MCP, најбоље праксе за безбедност, миграцију са SSE и практичне савете за изградњу ваших сопствених стриминг MCP апликација.

> **Поглед у будућност:** овај лекција описује Стриминг HTTP под **MCPspecifikacijom 2025-11-25**, где се сесија успоставља током `initialize` и везује са `Mcp-Session-Id` заглављем. Издање кандидата за верзију `2026-07-28` укида руковање и идентификатор сесије у потпуности, чинећи сваки захтев самосталним и рутирабилним ка било којој инстанци сервера без потребе за „липљивим“ сесијама. Погледајте [Шта се мења у MCP: Издање кандидата за верзију 2026-07-28](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md) за детаље.

## Транспортни механизми и стриминг у MCP

Одељак истражује различите транспортне механизме доступне у MCP и њихову улогу у омогућавању стриминга за комуникацију у реалном времену између клијената и сервера.

### Шта је транспортни механизам?

Транспортни механизам дефинише како се подаци размењују између клијента и сервера. MCP подржава више типова транспорта да одговарају различитим окружењима и захтевима:

- **stdio**: Стандардни улаз/излаз, погодан за локалне и CLI алате. Једноставан, али није погодан за веб или облак.
- **SSE (Server-Sent Events)**: Омогућава серверима да шаљу ажурирања у реалном времену ка клијентима преко HTTP. Добро за веб интерфејсе, али ограничено у скалабилности и флексибилности. Од MCP Спецификације 2025-06-18, самостални SSE транспорт је застарео и замењен транспортом "Стриминг HTTP".
- **Стриминг HTTP**: Модеран HTTP заснован транспорт за стриминг, који подржава обавештења и бољу скалабилност. Препоручује се за већину продукционих и облачних сценарија.

### Табела поређења

Погледајте табелу поређења испод да бисте разумели разлике између ових транспортних механизама:

| Транспорт        | Ажурирања у реалном времену | Стриминг | Скалабилност | Примена                   |
|------------------|-----------------------------|----------|--------------|---------------------------|
| stdio            | Не                          | Не       | Ниска        | Локални CLI алати         |
| SSE              | Да                          | Да       | Средња       | Веб, ажурирања у реалном времену |
| Стриминг HTTP    | Да                          | Да       | Висока       | Облак, више клијената     |

> **Савет:** Избор правог транспорта утиче на перформансе, скалабилност и корисничко искуство. **Стриминг HTTP** се препоручује за модерне, скалабилне и облакоприлагођене апликације.

Обратите пажњу на транспорте stdio и SSE који су приказани у претходним поглављима и како је стриминг HTTP транспорт који је покривен у овом поглављу.

## Стриминг: Концепти и мотивација

Разумевање основних концепта и мотивација иза стриминга је битно за имплементацију ефикасних система за комуникацију у реалном времену.

**Стриминг** је техника у мрежном програмирању која омогућава слање и примање података у малим, управљивим деловима или као низ догађаја, уместо да се чека целокупан одговор. Ово је посебно корисно за:

- Велике фајлове или скупове података.
- Ажурирања у реалном времену (нпр. чет, траке напретка).
- Дуго трајуће рачунање када желите да корисник буде обавештен.

Ево шта треба да знате о стримингу на високом нивоу:

- Подаци се испоручују постепено, не одједном.
- Клијент може да обрађује податке док пристижу.
- Смањује перципирану латенцију и побољшава корисничко искуство.

### Зашто користити стриминг?

Разлози за коришћење стриминга су следећи:

- Корисници добијају повратну информацију одмах, не само на крају
- Омогућава апликације у реалном времену и одзивне интерфејсе
- Ефикаснија употреба мрежних и рачунарских ресурса

### Једноставан пример: HTTP стриминг сервер и клијент

Ево једног једноставног примера како се стриминг може имплементирати:

#### Python

**Сервер (Python, користећи FastAPI и StreamingResponse):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**Клијент (Python, користећи requests):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

Овај пример демонстрира сервер који шаље низ порука клијенту како постају доступне, уместо да чека све поруке да буду спремне.

**Како функционише:**

- Сервер шаље сваку поруку када је спремна.
- Клијент прима и исписује сваки део како пристиже.

**Захтеви:**

- Сервер мора користити стриминг одговор (нпр. `StreamingResponse` у FastAPI).
- Клијент мора обрађивати одговор као стрим (`stream=True` у requests).
- Content-Type је обично `text/event-stream` или `application/octet-stream`.

#### Java

**Сервер (Java, користећи Spring Boot и Server-Sent Events):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**Клијент (Java, користећи Spring WebFlux WebClient):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**Напомене о имплементацији у Јави:**

- Користи реактивни стек Spring Boot-а са `Flux` за стриминг
- `ServerSentEvent` обезбеђује структурирани стриминг догађаја са типовима догађаја
- `WebClient` са `bodyToFlux()` омогућава реактивно конзумирање стрима
- `delayElements()` симулира време обраде између догађаја
- Догађаји могу имати типове (`info`, `result`) за боље руковање на клијенту

### Поређење: Класични стриминг насупрот MCP стриминга

Разлике између начина рада стриминга у „класичном“ облику и како ради у MCP могу се приказати овако:

| Карактеристика         | Класични HTTP стриминг        | MCP стриминг (Обавештења)        |
|-----------------------|-------------------------------|----------------------------------|
| Главни одговор        | Подељен на делове             | Јединствен, на крају              |
| Ажурирања напретка    | Слање као делови података      | Слање као обавештења             |
| Захтеви клијента       | Мора обрадити стрим           | Мора имплементирати обраду порука |
| Примена               | Велики фајлови, AI токени     | Прогрес, логови, повратне информације у реалном времену |

### Кључне разлике

Такође, ево неких кључних разлика:

- **Образац комуникације:**
  - Класични HTTP стриминг: Користи једноставно кодирање преноса делова
  - MCP стриминг: Користи структурирани систем обавештења са JSON-RPC протоколом

- **Формат поруке:**
  - Класични HTTP: Обични текст са новим линијама
  - MCP: Структурисани LoggingMessageNotification објекти са мета-подацима

- **Имплементација клијента:**
  - Класични HTTP: Једноставан клијент који обрађује стриминг одговоре
  - MCP: Напреднији клијент са обрадом различитих типова порука

- **Ажурирања напретка:**
  - Класични HTTP: Прогрес је део главног одговора
  - MCP: Прогрес се шаље као посебне обавештења, док главни одговор долази на крају

### Препоруке

Постоје неке ствари које препоручујемо у избору између класицног стриминга (као што смо показали горе користећи `/stream`) и стриминга преко MCP-а.

- **За једноставне потребе стриминга:** Класични HTTP стриминг је једноставнији за имплементацију и довољан за основне потребе.

- **За сложене, интерактивне апликације:** MCP стриминг пружа структуриранији приступ са богатијим метаподацима и раздвајањем између обавештења и крајњих резултата.

- **За AI апликације:** MCP систем обавештења је посебно користан за дуготрајне AI задатке где желите да кориснике информишете о напретку.

## Стриминг у MCP

Добро, до сада сте видели неке препоруке и поређења о разликама између класичног стриминга и MCP стриминга. Хајде да детаљније разгледамо како можете искористити стриминг у MCP.

Разумевање како стриминг функционише у оквиру MCP-а је битно за изградњу одзивних апликација које пружају повратне информације у реалном времену током дуготрајних операција.

У MCP, стриминг није слање главног одговора у деловима, већ слање **обавештења** клијенту док алат обрађује захтев. Та обавештења могу укључивати ажурирања напретка, логове или друге догађаје.

### Како функционише

Главни резултат се и даље шаље као један одговор. Међутим, обавештења се могу слати као посебне поруке током обраде и тако ажурирати клијента у реалном времену. Клијент мора бити у стању да обрађује и приказује ова обавештења.

## Шта је обавештење?

Рекли смо "Обавештење", шта то значи у контексту MCP-а?

Обавештење је порука послата са сервера ка клијенту да информише о напретку, статусу или другим догађајима током дуготрајне операције. Обавештења побољшавају транспарентност и корисничко искуство.

На пример, клијент треба да пошаље обавештење када је иницијални руковалац са сервером успешно завршен.

Обавештење изгледа овако као JSON порука:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

Обавештења припадају теми у MCP познатој као ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging).

> **Напомена о укидању:** кандидат за издање MCP спецификације `2026-07-28` означава Logging примитив као застарео у корист `stderr` за stdio транспорте и OpenTelemetry за структурирану видљивост. Логовање наставља да ради у `2025-11-25` и најмање годину дана након било какве формалне депрекације. Погледајте [Шта се мења у MCP: Издање кандидата 2026-07-28](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md).

Да би логовање функционисало, сервер треба да га омогући као функцију/могућност овако:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> У зависности од коришћеног SDK-а, логовање може бити подразумевано омогућено или га је потребно експлицитно укључити у конфигурацији сервера.

Постоје различити типови обавештења:

| Ниво       | Опис                          | Пример употребе               |
|------------|-------------------------------|------------------------------|
| debug      | Детаљне информације за отклањање грешака | Тачке уласка/изласка функције |
| info       | Опште информационе поруке    | Ажурирања напретка операција |
| notice     | Нормални али значајни догађаји | Промене конфигурације          |
| warning    | Упозорења                    | Коришћење застареле функције  |
| error      | Услов грешке                | Неуспеси операција           |
| critical   | Критични услови              | Неуспеси система             |
| alert      | Мора се одмах предузети акција | Откривена корупција података |
| emergency  | Систем је неисправан         | Потпуни пад система          |

## Имплементација обавештења у MCP

Да бисте имплементирали обавештења у MCP, треба да подесите и серверску и клијентску страну да рукују ажурирањима у реалном времену. Ово омогућава вашој апликацији да пружа тренутне повратне информације корисницима током дуготрајних операција.

### Серверска страна: слање обавештења

Почнимо са серверском страном. У MCP-у, дефинишете алате који могу слати обавештења током обраде захтева. Сервер користи контекст објекат (обично `ctx`) да шаље поруке клијенту.

#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

У наведеном примеру, алат `process_files` шаље три обавештења клијенту док обрађује сваки фајл. Метод `ctx.info()` се користи за слање информативних порука.

Додатно, да бисте омогућили обавештења, уверите се да ваш сервер користи стриминг транспорт (попут `streamable-http`) и да ваш клијент има имплементирану обраду порука за обавештења. Ево како да подесите сервер за `streamable-http` транспорт:

```python
mcp.run(transport="streamable-http")
```

#### .NET

```csharp
[Tool("A tool that sends progress notifications")]
public async Task<TextContent> ProcessFiles(string message, ToolContext ctx)
{
    await ctx.Info("Processing file 1/3...");
    await ctx.Info("Processing file 2/3...");
    await ctx.Info("Processing file 3/3...");
    return new TextContent
    {
        Type = "text",
        Text = $"Done: {message}"
    };
}
```

У овом .NET примеру, алат `ProcessFiles` је означен атрибутом `Tool` и шаље три обавештења клијенту током обраде сваког фајла. Метод `ctx.Info()` се користи за слање информативних порука.

Да бисте омогућили обавештења у вашем .NET MCP серверу, осигурајте да користите стриминг транспорт:

```csharp
var builder = McpBuilder.Create();
await builder
    .UseStreamableHttp() // Enable streamable HTTP transport
    .Build()
    .RunAsync();
```

### Клијентска страна: пријем обавештења

Клијент мора имати имплементирану обраду порука која обрађује и приказује обавештења како пристижу.

#### Python

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

У наведеној коди, функција `message_handler` проверава да ли порука представља обавештење. Ако јесте, исписује обавештење, иначе је обрађује као редован серверски одговор. Такође, `ClientSession` је иницијализован са `message_handler` да обрађује пристижућа обавештења.

#### .NET

```csharp
// Define a message handler
void MessageHandler(IJsonRpcMessage message)
{
    if (message is ServerNotification notification)
    {
        Console.WriteLine($"NOTIFICATION: {notification}");
    }
    else
    {
        Console.WriteLine($"SERVER MESSAGE: {message}");
    }
}

// Create and use a client session with the message handler
var clientOptions = new ClientSessionOptions
{
    MessageHandler = MessageHandler,
    LoggingCallback = (level, message) => Console.WriteLine($"[{level}] {message}")
};

using var client = new ClientSession(readStream, writeStream, clientOptions);
await client.InitializeAsync();

// Now the client will process notifications through the MessageHandler
```

У овом .NET примеру, функција `MessageHandler` проверава да ли је порука обавештење. Ако јесте, исписује обавештење, иначе је обрађује као редован серверски одговор. `ClientSession` је иницијализован са обрадом порука преко `ClientSessionOptions`.

Да бисте омогућили обавештења, уверите се да ваш сервер користи стриминг транспорт (попут `streamable-http`) и да ваш клијент ради обраду обавештења.

## Обавештења о напретку и сценарији

Овај одељак објашњава концепт обавештавања о напретку у MCP, зашто је важно и како их имплементирати користећи Стриминг HTTP. Такође, садржи практичан задатак за учвршћивање знања.

Обавештења о напретку су поруке у реалном времену које сервер шаље клијенту током дуготрајних операција. Уместо да се чека да целокупан процес буде завршен, сервер ажурира клијента о тренутном статусу. Ово побољшава транспарентност, корисничко искуство и олакшава отклањање грешака.

**Пример:**

```text

"Processing document 1/10"
"Processing document 2/10"
...
"Processing complete!"

```

### Зашто користити обавештења о напретку?

Обавештења о напретку су битна из више разлога:

- **Боље корисничко искуство:** Корисници виде ажурирања током рада, не само на крају.
- **Повратна информација у реалном времену:** Клијенти могу да приказују траке напретка или логове, чинећи апликацију одзивном.
- **Лакше отклањање грешака и надзор:** Програмери и корисници могу видети где процес може да успорава или стане.

### Како имплементирати обавештења о напретку

Ево како можете имплементирати обавештења о напретку у MCP:

- **На серверу:** Користите `ctx.info()` или `ctx.log()` да шаљете обавештења како се сваки предмет обрађује. Ово шаље поруку клијенту пре него што главни резултат буде спреман.
- **На клијенту:** Имплементирајте обраду порука која слуша и приказује обавештења како пристижу. Ова обрада разликује обавештења од коначног резултата.

**Пример сервера:**


#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    for i in range(1, 11):
        await ctx.info(f"Processing document {i}/10")
    await ctx.info("Processing complete!")
    return TextContent(type="text", text=f"Done: {message}")
```

**Пример клијента:**

#### Python

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)
```

## Безбедносне напомене

Када имплементирате MCP сервере са HTTP-транспортима, безбедност постаје од пресудног значаја и захтева пажљиво разматрање више вектора напада и заштитних механизама.

### Преглед

Безбедност је критична када се MCP сервери излажу преко HTTP-а. Стриминг HTTP уводи нове површине за напад и захтева пажљиву конфигурацију.

### Кључне тачке

- **Валидација Origin заглавља**: Увек валидирајте `Origin` заглавље да бисте спречили DNS ребајндинг нападе.
- **Везивање на localhost**: За локални развој, повежите сервере на `localhost` да бисте спречили изложеност јавном интернету.
- **Аутентификација**: Имплементирајте аутентификацију (нпр. API кључеве, OAuth) за производна окружења.
- **CORS**: Конфигуришите политике Cross-Origin Resource Sharing (CORS) да ограничите приступ.
- **HTTPS**: Користите HTTPS у производњи за шифровање саобраћаја.

### Најбоље праксе

- Никад не верујте улазним захтевима без верификације.
- Логовајте и пратите сав приступ и грешке.
- Редовно ажурирајте зависности да бисте закрпили безбедносне рањивости.

### Изазови

- Усклађивање безбедности са лакоћом развоја
- Осигуравање компатибилности са различитим окружењима клијената

## Надоградња са SSE на Стриминг HTTP

За апликације које тренутно користе Server-Sent Events (SSE), миграција на Стриминг HTTP пружа побољшане могућности и бољу одрживост MCP имплементација на дуги рок.

### Зашто надоградити?

Постоје два убедљива разлога за надоградњу са SSE на Стриминг HTTP:

- Стриминг HTTP нуди бољу скалабилност, компатибилност и богатију подршку за обавештења у односу на SSE.
- То је препоручени транспорт за нове MCP апликације.

### Кораци миграције

Ево како можете мигрирати са SSE на Стриминг HTTP у вашим MCP апликацијама:

- **Ажурирајте серверски код** да користи `transport="streamable-http"` у `mcp.run()`.
- **Ажурирајте клијентски код** да користи `streamablehttp_client` уместо SSE клијента.
- **Имплементирајте обраду порука** у клијенту за обраду обавештења.
- **Тестирајте компатибилност** са постојећим алатима и радним токовима.

### Одржавање компатибилности

Препоручује се одржавање компатибилности са постојећим SSE клијентима током процеса миграције. Ево неких стратегија:

- Можете подржати и SSE и Стриминг HTTP тако што ћете их покренути на различитим ендпоинтовима.
- Постепено мигрирајте клијенте на нови транспорт.

### Изазови

Осигурајте да током миграције адресирате следеће изазове:

- Осигурање да су сви клијенти ажурирани
- Руковање разликама у достави обавештења
