# پخش HTTPS با پروتکل مدل کانتکست (MCP)

این فصل راهنمایی جامع برای پیاده‌سازی پخش امن، مقیاس‌پذیر و بلادرنگ با استفاده از پروتکل مدل کانتکست (MCP) از طریق HTTPS فراهم می‌کند. این فصل انگیزه برای پخش، مکانیزم‌های حمل‌ونقل موجود، چگونگی پیاده‌سازی HTTP قابل پخش در MCP، بهترین شیوه‌های امنیتی، مهاجرت از SSE و راهنمای عملی برای ساخت برنامه‌های پخش MCP را پوشش می‌دهد.

> **نگاهی به آینده:** این درس HTTP قابل پخش را تحت **مشخصات MCP ۲۰۲۵-۱۱-۲۵** شرح می‌دهد، جایی که جلسه‌ای هنگام `initialize` ایجاد و با هدر `Mcp-Session-Id` مشخص می‌شود. نامزد نسخه ۲۰۲۶-۰۷-۲۸، دست دادن و شناسه جلسه را کاملاً حذف می‌کند، و هر درخواست را خودکفا و قابل مسیریابی به هر نمونه سرور بدون جلسات چسبنده می‌سازد. برای جزئیات بیشتر به [چه تغییراتی در MCP: نامزد انتشار ۲۰۲۶-۰۷-۲۸](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md) مراجعه کنید.

## مکانیزم‌های حمل‌ونقل و پخش در MCP

این بخش مکانیزم‌های مختلف حمل‌ونقل موجود در MCP و نقش آن‌ها در فعال‌سازی قابلیت‌های پخش برای ارتباط بلادرنگ بین مشتریان و سرورها را بررسی می‌کند.

### مکانیزم حمل‌ونقل چیست؟

مکانیزم حمل‌ونقل نحوه تبادل داده بین مشتری و سرور را تعریف می‌کند. MCP از انواع مختلف حمل‌ونقل پشتیبانی می‌کند تا با محیط‌ها و نیازهای متفاوت سازگار باشد:

- **stdio**: ورودی/خروجی استاندارد، مناسب برای ابزارهای محلی و مبتنی بر خط فرمان. ساده اما برای وب یا فضای ابری مناسب نیست.
- **SSE (رویدادهای ارسالی سرور)**: اجازه می‌دهد سرورها به‌روزرسانی‌های بلادرنگ را از طریق HTTP به مشتریان ارسال کنند. برای رابط‌های وب مناسب است، اما محدودیت‌هایی در مقیاس‌پذیری و انعطاف‌پذیری دارد. طبق مشخصات MCP ۲۰۲۵-۰۶-۱۸، حمل‌ونقل SSE مستقل منسوخ شده و با حمل‌ونقل "HTTP قابل پخش" جایگزین شده است.
- **HTTP قابل پخش**: حمل‌ونقل مبتنی بر HTTP مدرن، با پشتیبانی از اعلان‌ها و مقیاس‌پذیری بهتر. برای اکثر سناریوهای تولید و ابری توصیه می‌شود.

### جدول مقایسه

برای درک تفاوت‌های این مکانیزم‌های حمل‌ونقل، جدول مقایسه زیر را مشاهده کنید:

| مکانیزم حمل‌ونقل | به‌روزرسانی بلادرنگ | پخش | مقیاس‌پذیری | مورد استفاده                |
|-------------------|------------------|-----------|-------------|-------------------------|
| stdio             | نه               | نه        | کم          | ابزارهای محلی خط فرمان    |
| SSE               | بله              | بله       | متوسط       | وب، به‌روزرسانی‌های بلادرنگ  |
| HTTP قابل پخش     | بله              | بله       | بالا        | ابر، چند مشتری     |

> **نکته:** انتخاب مکانیزم حمل‌ونقل مناسب بر عملکرد، مقیاس‌پذیری و تجربه کاربری تأثیر می‌گذارد. **HTTP قابل پخش** برای برنامه‌های مدرن، مقیاس‌پذیر و مناسب ابر توصیه می‌شود.

توجه کنید که حمل‌ونقل‌های stdio و SSE که در فصول قبلی نشان داده شدند و چگونه HTTP قابل پخش مکانیزم حمل‌ونقلی است که در این فصل پوشش داده می‌شود.

## پخش: مفاهیم و انگیزه

درک مفاهیم و انگیزه‌های بنیادی پشت پخش برای پیاده‌سازی سیستم‌های ارتباطی بلادرنگ مؤثر ضروری است.

**پخش** تکنیکی در برنامه‌نویسی شبکه است که اجازه می‌دهد داده‌ها به صورت قطعات کوچک و قابل مدیریت یا به شکل دنباله‌ای از رویدادها ارسال و دریافت شوند، به جای اینکه منتظر پاسخ کامل باشیم. این موضوع به‌ویژه برای موارد زیر مفید است:

- فایل‌ها یا مجموعه داده‌های بزرگ.
- به‌روزرسانی‌های بلادرنگ (مثلاً چت، نوارهای پیشرفت).
- محاسبات طولانی‌مدتی که می‌خواهید کاربر را مطلع نگه دارید.

در اینجا نکات کلیدی برای فهم پخش به سطح بالا آمده است:

- داده به تدریج ارسال می‌شود، نه همه یکجا.
- مشتری می‌تواند داده را هنگام دریافت پردازش کند.
- کاهش تأخیر ادراک شده و بهبود تجربه کاربری.

### چرا از پخش استفاده کنیم؟

دلایل استفاده از پخش عبارتند از:

- کاربران فوراً بازخورد می‌گیرند، نه فقط در پایان.
- امکان‌پذیر کردن برنامه‌های بلادرنگ و رابط‌های پاسخگو.
- استفاده کارآمدتر از منابع شبکه و محاسبات.

### مثال ساده: سرور و مشتری پخش HTTP

در اینجا یک مثال ساده از نحوه پیاده‌سازی پخش آمده است:

#### پایتون

**سرور (پایتون، با استفاده از FastAPI و StreamingResponse):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**مشتری (پایتون، با استفاده از requests):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

این مثال نشان می‌دهد که چگونه سرور پیام‌هایی را به صورت دنباله‌ای به مشتری ارسال می‌کند به محض اینکه آماده شدند، به‌جای اینکه منتظر همه پیام‌ها باشد.

**نحوه کار:**

- سرور هر پیام را هنگام آماده شدن بیرون می‌دهد.
- مشتری هر بخش را هنگام دریافت چاپ می‌کند.

**نیازمندی‌ها:**

- سرور باید پاسخ پخش‌پذیر (مثلاً `StreamingResponse` در FastAPI) استفاده کند.
- مشتری باید پاسخ را به صورت جریان پردازش کند (`stream=True` در requests).
- نوع محتوا معمولاً `text/event-stream` یا `application/octet-stream` است.

#### جاوا

**سرور (جاوا، با استفاده از Spring Boot و Server-Sent Events):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**مشتری (جاوا، با استفاده از Spring WebFlux WebClient):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**یادداشت‌های پیاده‌سازی جاوا:**

- از استک واکنشی Spring Boot با `Flux` برای پخش استفاده می‌کند.
- `ServerSentEvent` پخش رویداد ساختاریافته با انواع رویداد را فراهم می‌کند.
- `WebClient` با `bodyToFlux()` مصرف پخش واکنشی را میسر می‌کند.
- `delayElements()` زمان پردازش بین رویدادها را شبیه‌سازی می‌کند.
- رویدادها می‌توانند انواعی مثل (`info`، `result`) داشته باشند برای مدیریت بهتر مشتری.

### مقایسه: پخش کلاسیک در مقابل پخش MCP

تفاوت‌های چگونگی کارکرد پخش به روش «کلاسیک» در مقابل چگونه کارکرد آن در MCP را می‌توان به این صورت نمایش داد:

| ویژگی                 | پخش کلاسیک HTTP             | پخش MCP (اطلاع‌رسانی‌ها)      |
|------------------------|-----------------------------|-----------------------------------|
| پاسخ اصلی             | قطعه‌ای (Chunked)           | واحد، در پایان                   |
| به‌روزرسانی‌های پیشرفت| به صورت قطعات داده ارسال می‌شود | به صورت اطلاع‌رسانی‌ها ارسال می‌شود |
| نیازهای مشتری          | باید جریان را پردازش کند    | باید هندلر پیام پیاده‌سازی کند  |
| مورد استفاده           | فایل‌های بزرگ، جریان‌های توکن AI | پیشرفت، لاگ‌ها، بازخورد بلادرنگ  |

### تفاوت‌های کلیدی مشاهده شده

علاوه بر این، برخی تفاوت‌های کلیدی عبارتند از:

- **الگوی ارتباطی:**  
  - پخش کلاسیک HTTP: از کدگذاری انتقال ساده قطعه‌ای برای ارسال داده استفاده می‌کند.  
  - پخش MCP: از سیستم اطلاع‌رسانی ساختاریافته با پروتکل JSON-RPC بهره می‌برد.  

- **فرمت پیام:**  
  - پخش کلاسیک HTTP: قطعات متنی ساده با خطوط جدید.  
  - MCP: اشیاء ساختاریافته LoggingMessageNotification با فراداده‌ها.  

- **پیاده‌سازی مشتری:**  
  - پخش کلاسیک HTTP: مشتری ساده که پاسخ‌های پخش را پردازش می‌کند.  
  - MCP: مشتری پیچیده‌تر با هندلر پیام برای پردازش انواع مختلف پیام‌ها.  

- **به‌روزرسانی‌های پیشرفت:**  
  - پخش کلاسیک HTTP: پیشرفت بخشی از جریان پاسخ اصلی است.  
  - MCP: پیشرفت از طریق پیام‌های اطلاع‌رسانی جداگانه ارسال می‌شود در حالی که پاسخ اصلی در پایان می‌آید.  

### توصیه‌ها

برخی توصیه‌ها درباره انتخاب بین پیاده‌سازی پخش کلاسیک (به عنوان نقطه پایانی که با `/stream` نشان داده شد) در مقابل انتخاب پخش از طریق MCP وجود دارد.

- **برای نیازهای پخش ساده:** پخش کلاسیک HTTP ساده‌تر است و برای نیازهای پایه پخش کافی است.

- **برای برنامه‌های پیچیده و تعاملی:** پخش MCP رویکرد ساختاریافته‌تری با متادیتای غنی‌تر و جداسازی بین اطلاع‌رسانی‌ها و نتایج نهایی ارائه می‌دهد.

- **برای برنامه‌های هوش مصنوعی:** سیستم اطلاع‌رسانی MCP به ویژه برای وظایف هوش مصنوعی طولانی‌مدت که می‌خواهید کاربران از پیشرفت مطلع شوند، مفید است.

## پخش در MCP

خب، پس تا الان چند توصیه و مقایسه درباره تفاوت‌های پخش کلاسیک و پخش در MCP دیده‌اید. حالا دقیق‌تر می‌رویم ببینیم چگونه می‌توانید از پخش در MCP بهره ببرید.

درک چگونگی کارکرد پخش در چارچوب MCP برای ساخت برنامه‌های پاسخگو که بازخورد بلادرنگ به کاربران در طول عملیات طولانی ارائه می‌کنند، ضروری است.

در MCP، پخش به معنای ارسال پاسخ اصلی به صورت بخش‌بندی شده نیست، بلکه ارسال **اطلاع‌رسانی‌ها** به مشتری است در حالی که ابزار در حال پردازش درخواست است. این اطلاع‌رسانی‌ها می‌توانند شامل به‌روزرسانی پیشرفت، لاگ‌ها یا رویدادهای دیگر باشند.

### نحوه کار

نتیجه اصلی همچنان به صورت یک پاسخ واحد ارسال می‌شود. با این حال، اطلاع‌رسانی‌ها می‌توانند به صورت پیام‌های جداگانه در طول پردازش ارسال شوند و بدین ترتیب مشتری را در زمان واقعی به‌روزرسانی کنند. مشتری باید توانایی دریافت و نمایش این اطلاع‌رسانی‌ها را داشته باشد.

## اطلاع‌رسانی چیست؟

گفتیم «اطلاع‌رسانی»، در زمینه MCP به چه معناست؟

اطلاع‌رسانی پیامی است که از سرور به مشتری ارسال می‌شود تا درباره پیشرفت، وضعیت یا رویدادهای دیگر در طول یک عملیات طولانی‌مدت اطلاع دهد. اطلاع‌رسانی‌ها شفافیت و تجربه کاربری را بهبود می‌بخشند.

برای مثال، انتظار می‌رود مشتری یک اطلاع‌رسانی ارسال کند وقتی نخستین دست دادن با سرور انجام شده است.

یک اطلاع‌رسانی به این شکل به صورت پیام JSON است:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

اطلاع‌رسانی‌ها به یک موضوع در MCP تعلق دارند که به عنوان ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging) شناخته می‌شود.

> **اطلاعیه منسوخ شدن:** نامزد نسخه مشخصات MCP ۲۰۲۶-۰۷-۲۸، اصل Logging را به نفع `stderr` برای حمل‌ونقل stdio و OpenTelemetry برای رصد ساختاریافته منسوخ اعلام می‌کند. Logging در نسخه ۲۰۲۵-۱۱-۲۵ و حداقل یک سال پس از هر منسوخ‌سازی رسمی به کار خود ادامه می‌دهد. برای جزئیات بیشتر به [چه تغییراتی در MCP: نامزد انتشار ۲۰۲۶-۰۷-۲۸](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md) مراجعه کنید.

برای فعال‌سازی لاگینگ، سرور باید آن را به عنوان ویژگی/قابلیت به این صورت فعال کند:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> بسته به SDK مورد استفاده، ممکن است لاگینگ به طور پیش‌فرض فعال باشد یا نیاز باشد که به طور صریح در تنظیمات سرور آن را فعال کنید.

انواع مختلفی از اطلاع‌رسانی‌ها وجود دارد:

| سطح      | توضیح                       | مثال مورد استفاده              |
|-----------|---------------------------|-------------------------------|
| debug     | اطلاعات دقیق اشکال‌زدایی   | نقاط ورود/خروج توابع          |
| info      | پیام‌های اطلاعاتی عمومی     | به‌روزرسانی پیشرفت عملیات    |
| notice    | رویدادهای معمول ولی مهم    | تغییرات پیکربندی              |
| warning   | شرایط هشدار                | استفاده از ویژگی منسوخ شده    |
| error     | شرایط خطا                  | شکست عملیات                   |
| critical  | شرایط بحرانی              | شکست‌های اجزای سیستم         |
| alert     | باید فوراً اقدام شود       | تشخیص فساد داده               |
| emergency | سیستم غیرقابل استفاده است | خرابی کامل سیستم             |

## پیاده‌سازی اطلاع‌رسانی‌ها در MCP

برای پیاده‌سازی اطلاع‌رسانی‌ها در MCP، باید هر دو سمت سرور و مشتری را برای دریافت به‌روزرسانی‌های بلادرنگ تنظیم کنید. این امکان را می‌دهد که برنامه شما در طول عملیات طولانی بازخورد فوری به کاربران ارائه دهد.

### سمت سرور: ارسال اطلاع‌رسانی‌ها

با سمت سرور شروع کنیم. در MCP، شما ابزارهایی را تعریف می‌کنید که می‌توانند هنگام پردازش درخواست‌ها اطلاع‌رسانی ارسال کنند. سرور از شیء کانتکست (معمولاً `ctx`) برای ارسال پیام‌ها به مشتری استفاده می‌کند.

#### پایتون

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

در مثال قبلی، ابزار `process_files` سه اطلاع‌رسانی به مشتری ارسال می‌کند در حالی که هر فایل را پردازش می‌کند. متد `ctx.info()` برای ارسال پیام‌های اطلاعاتی استفاده می‌شود.

علاوه بر این، برای فعال کردن اطلاع‌رسانی‌ها، اطمینان حاصل کنید که سرور شما از حمل‌ونقل پخش‌پذیر (مثلاً `streamable-http`) استفاده می‌کند و مشتری هندلر پیام برای پردازش اطلاع‌رسانی‌ها پیاده‌سازی کرده است. در اینجا نحوه راه‌اندازی سرور برای استفاده از حمل‌ونقل `streamable-http` آمده است:

```python
mcp.run(transport="streamable-http")
```

#### .NET

```csharp
[Tool("A tool that sends progress notifications")]
public async Task<TextContent> ProcessFiles(string message, ToolContext ctx)
{
    await ctx.Info("Processing file 1/3...");
    await ctx.Info("Processing file 2/3...");
    await ctx.Info("Processing file 3/3...");
    return new TextContent
    {
        Type = "text",
        Text = $"Done: {message}"
    };
}
```

در این مثال .NET، ابزار `ProcessFiles` با صفت `Tool` تزئین شده و سه اطلاع‌رسانی به مشتری ارسال می‌کند در حالی که هر فایل را پردازش می‌کند. متد `ctx.Info()` برای ارسال پیام‌های اطلاعاتی استفاده می‌شود.

برای فعال کردن اطلاع‌رسانی‌ها در سرور MCP خود در .NET، اطمینان حاصل کنید که از حمل‌ونقل پخش‌پذیر استفاده می‌کنید:

```csharp
var builder = McpBuilder.Create();
await builder
    .UseStreamableHttp() // Enable streamable HTTP transport
    .Build()
    .RunAsync();
```

### سمت مشتری: دریافت اطلاع‌رسانی‌ها

مشتری باید هندلر پیام را پیاده‌سازی کند تا اطلاع‌رسانی‌ها را هنگام دریافت پردازش و نمایش دهد.

#### پایتون

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

در کد قبلی، تابع `message_handler` بررسی می‌کند که آیا پیام ورودی یک اطلاع‌رسانی است یا خیر. اگر اطلاع‌رسانی باشد، آن را چاپ می‌کند؛ در غیر این صورت، آن را به عنوان پیام معمولی سرور پردازش می‌کند. همچنین توجه کنید که چگونه `ClientSession` با `message_handler` برای مدیریت اطلاع‌رسانی‌های ورودی مقداردهی اولیه شده است.

#### .NET

```csharp
// Define a message handler
void MessageHandler(IJsonRpcMessage message)
{
    if (message is ServerNotification notification)
    {
        Console.WriteLine($"NOTIFICATION: {notification}");
    }
    else
    {
        Console.WriteLine($"SERVER MESSAGE: {message}");
    }
}

// Create and use a client session with the message handler
var clientOptions = new ClientSessionOptions
{
    MessageHandler = MessageHandler,
    LoggingCallback = (level, message) => Console.WriteLine($"[{level}] {message}")
};

using var client = new ClientSession(readStream, writeStream, clientOptions);
await client.InitializeAsync();

// Now the client will process notifications through the MessageHandler
```

در این مثال .NET، تابع `MessageHandler` بررسی می‌کند که آیا پیام ورودی یک اطلاع‌رسانی است یا خیر. اگر اطلاع‌رسانی باشد، آن را چاپ می‌کند؛ در غیر این صورت، آن را به عنوان پیام معمولی سرور پردازش می‌کند. `ClientSession` از طریق `ClientSessionOptions` با هندلر پیام مقداردهی اولیه شده است.

برای فعال کردن اطلاع‌رسانی‌ها، اطمینان حاصل کنید که سرور شما از حمل‌ونقل پخش‌پذیر (مانند `streamable-http`) استفاده می‌کند و مشتری هندلر پیام برای پردازش اطلاع‌رسانی پیاده‌سازی کرده است.

## اطلاع‌رسانی‌های پیشرفت و سناریوها

این بخش مفهوم اطلاع‌رسانی پیشرفت در MCP، اهمیت آن و نحوه پیاده‌سازی آن با استفاده از HTTP قابل پخش را توضیح می‌دهد. همچنین یک تمرین عملی برای تقویت درکتان موجود است.

اطلاع‌رسانی‌های پیشرفت پیام‌های بلادرنگی هستند که از سرور به مشتری در طول عملیات طولانی ارسال می‌شوند. به جای اینکه منتظر پایان کامل عملیات باشید، سرور وضعیت جاری را به مشتری اطلاع می‌دهد. این موضوع شفافیت، تجربه کاربری و عیب‌یابی را آسان‌تر می‌کند.

**مثال:**

```text

"Processing document 1/10"
"Processing document 2/10"
...
"Processing complete!"

```

### چرا از اطلاع‌رسانی‌های پیشرفت استفاده کنیم؟

اطلاع‌رسانی‌های پیشرفت برای چند دلیل حیاتی هستند:

- **تجربه کاربری بهتر:** کاربران هنگام پیشرفت کار به‌روزرسانی‌ها را مشاهده می‌کنند، نه فقط در پایان.
- **بازخورد بلادرنگ:** مشتری‌ها می‌توانند نوارهای پیشرفت یا لاگ‌ها را نمایش دهند و برنامه پاسخگو به نظر برسد.
- **عیب‌یابی و پایش آسان‌تر:** توسعه‌دهندگان و کاربران می‌توانند ببینند کجا فرآیند ممکن است کند یا متوقف شده باشد.

### چگونه اطلاع‌رسانی‌های پیشرفت را پیاده‌سازی کنیم

در اینجا نحوه پیاده‌سازی اطلاع‌رسانی‌های پیشرفت در MCP آمده است:

- **در سمت سرور:** از `ctx.info()` یا `ctx.log()` برای ارسال اطلاع‌رسانی‌ها هنگام پردازش هر عنصر استفاده کنید. این پیام‌ها قبل از آماده شدن نتیجه اصلی به مشتری ارسال می‌شوند.
- **در سمت مشتری:** هندلر پیام پیاده‌سازی کنید که به اطلاع‌رسانی‌ها گوش دهد و آن‌ها را نمایش دهد. این هندلر اطلاع‌رسانی‌ها را از نتیجه نهایی تمایز می‌دهد.

**مثال سرور:**


#### پایتون

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    for i in range(1, 11):
        await ctx.info(f"Processing document {i}/10")
    await ctx.info("Processing complete!")
    return TextContent(type="text", text=f"Done: {message}")
```

**مثال مشتری:**

#### پایتون

## ملاحظات امنیتی

### مرور کلی

- **CORS**: سیاست‌های اشتراک منابع بین مبدأها (CORS) را برای محدود کردن دسترسی پیکربندی کنید.
- **HTTPS**: در محیط تولید از HTTPS برای رمزنگاری ترافیک استفاده کنید.

- به صورت منظم وابستگی‌ها را به‌روزرسانی کنید تا آسیب‌پذیری‌های امنیتی برطرف شود.

## ارتقا از SSE به HTTP قابل پخش

برای برنامه‌هایی که در حال حاضر از Server-Sent Events (SSE) استفاده می‌کنند، مهاجرت به HTTP قابل پخش امکانات بهتر و پایداری بلندمدت بهتری را برای پیاده‌سازی‌های MCP فراهم می‌کند.

### چرا ارتقا؟

دو دلیل قوی برای ارتقا از SSE به HTTP قابل پخش وجود دارد:

- HTTP قابل پخش مقیاس‌پذیری، سازگاری و پشتیبانی اعلاتر از اعلان‌ها را نسبت به SSE ارائه می‌دهد.
- این انتقال برای برنامه‌های جدید MCP توصیه شده است.

### مراحل مهاجرت

در اینجا نحوه مهاجرت از SSE به HTTP قابل پخش در برنامه‌های MCP شما آمده است:

- **به‌روزرسانی کد سرور** برای استفاده از `transport="streamable-http"` در `mcp.run()`.
- **به‌روزرسانی کد مشتری** برای استفاده از `streamablehttp_client` به جای کلاینت SSE.
- **پیاده‌سازی یک هندلر پیام** در مشتری برای پردازش اعلان‌ها.
- **تست سازگاری** با ابزارها و فرآیندهای موجود.

### حفظ سازگاری

توصیه می‌شود در فرآیند مهاجرت سازگاری با کلاینت‌های SSE موجود را حفظ کنید. در اینجا چند راهکار آمده است:

- می‌توانید هر دو پروتکل SSE و HTTP قابل پخش را با اجرای هر کدام در نقاط انتهایی مختلف پشتیبانی کنید.
- تدریجی کلاینت‌ها را به انتقال جدید مهاجرت دهید.

### چالش‌ها

اطمینان حاصل کنید که در طول مهاجرت چالش‌های زیر را برطرف می‌کنید:

- اطمینان از به‌روزرسانی همه کلاینت‌ها
- مدیریت تفاوت‌ها در تحویل اعلان‌ها

3. اجرای هر دو سرور و مشتری را تست کنید و اعلان‌ها را مشاهده نمایید.
