# HTTPS سٹریمنگ ماڈل کانٹیکسٹ پروٹوکول (MCP) کے ساتھ

یہ باب HTTPS کا استعمال کرتے ہوئے ماڈل کانٹیکسٹ پروٹوکول (MCP) کے ساتھ محفوظ، توسیع پذیر، اور ریئل ٹائم سٹریمنگ کو نافذ کرنے کے لئے مکمل رہنمائی فراہم کرتا ہے۔ اس میں سٹریمنگ کی ترغیب، دستیاب ٹرانسپورٹ میکانزم، MCP میں اسٹریم ایبل HTTP کو کیسے نافذ کیا جائے، سیکیورٹی کی بہترین مشقیں، SSE سے مائیگریشن، اور اپنے خود کے سٹریمنگ MCP ایپلیکیشنز بنانے کی عملی رہنمائی شامل ہے۔

> **آگے کی نظر:** اس سبق میں Streamable HTTP کو **MCP Specification 2025-11-25** کے تحت بیان کیا گیا ہے، جہاں سیشن `initialize` کے دوران قائم کیا جاتا ہے اور ایک `Mcp-Session-Id` ہیڈر کے ساتھ پن کیا جاتا ہے۔ `2026-07-28` کا ریلیز کینیڈیٹ ہینڈشیک اور سیشن آئی ڈی کو مکمل طور پر ہٹا دیتا ہے، جس سے ہر درخواست خود مختار اور کسی بھی سرور انسٹینس کو روٹ کی جا سکتی ہے بغیر اسٹیکی سیشنز کے۔ تفصیلات کے لئے دیکھیں [MCP میں کیا تبدیلیاں ہو رہی ہیں: 2026-07-28 ریلیز کینیڈیٹ](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md)۔

## MCP میں ٹرانسپورٹ میکانزم اور سٹریمنگ

یہ سیکشن MCP میں دستیاب مختلف ٹرانسپورٹ میکانزم اور کلائنٹس اور سرورز کے درمیان ریئل ٹائم مواصلات کے لئے سٹریمنگ کی صلاحیتیں فراہم کرنے میں ان کے کردار کو دریافت کرتا ہے۔

### ٹرانسپورٹ میکانزم کیا ہے؟

ٹرانسپورٹ میکانزم یہ بتاتا ہے کہ کس طرح ڈیٹا کلائنٹ اور سرور کے درمیان تبادلہ کیا جاتا ہے۔ MCP مختلف ماحول اور ضروریات کے مطابق متعدد ٹرانسپورٹ اقسام کی حمایت کرتا ہے:

- **stdio**: معیاری ان پٹ/آؤٹ پٹ، مقامی اور کمانڈ لائن ٹولز کے لئے موزوں۔ سادہ لیکن ویب یا کلاؤڈ کے لئے موزوں نہیں۔
- **SSE (سرور سے بھیجے گئے ایونٹس)**: سرور کو کلائنٹس کو HTTP کے ذریعے ریئل ٹائم اپڈیٹس بھیجنے کی اجازت دیتا ہے۔ ویب UIs کے لئے اچھا، لیکن توسیع پذیری اور لچک میں محدود۔ MCP Specification 2025-06-18 کے مطابق، اسٹینڈ الون SSE ٹرانسپورٹ کو ترک کر دیا گیا ہے اور اسے "Streamable HTTP" ٹرانسپورٹ سے تبدیل کر دیا گیا ہے۔
- **Streamable HTTP**: جدید HTTP پر مبنی سٹریمنگ ٹرانسپورٹ، نوٹیفیکیشنز اور بہتر توسیع پذیری کی حمایت کے ساتھ۔ زیادہ تر پروڈکشن اور کلاؤڈ حالات کے لئے تجویز کردہ۔

### موازنہ جدول

نیچے دیے گئے موازنہ جدول پر نظر ڈال کر ان ٹرانسپورٹ میکانزم کے درمیان فرق کو سمجھیں:

| ٹرانسپورٹ         | ریئل ٹائم اپڈیٹس | سٹریمنگ | توسیع پذیری | استعمال کا کیس                |
|-------------------|------------------|-----------|-------------|-------------------------|
| stdio             | نہیں             | نہیں      | کم          | مقامی CLI ٹولز         |
| SSE               | ہاں              | ہاں       | درمیانہ     | ویب، ریئل ٹائم اپڈیٹس  |
| Streamable HTTP   | ہاں              | ہاں       | زیادہ       | کلاؤڈ، ملٹی کلائنٹ     |

> **ٹپ:** صحیح ٹرانسپورٹ کا انتخاب کارکردگی، توسیع پذیری، اور یوزر تجربے پر اثر انداز ہوتا ہے۔ **Streamable HTTP** جدید، توسیع پذیر، اور کلاؤڈ تیار ایپلیکیشنز کے لئے تجویز کردہ ہے۔

نوٹ کریں کہ پچھلے ابواب میں آپ کو دکھائے گئے ٹرانسپورٹ stdio اور SSE ہیں اور اس باب میں جو کور کیا گیا ہے وہ streamable HTTP ہے۔

## سٹریمنگ: تصورات اور ترغیب

سٹریمنگ کے بنیادی تصورات اور محرکات کو سمجھنا مؤثر ریئل ٹائم کمیونیکیشن سسٹمز کو نافذ کرنے کے لیے ضروری ہے۔

**سٹریمنگ** ایک نیٹ ورک پروگرامنگ کی تکنیک ہے جو ڈیٹا کو چھوٹے، قابلِ انتظام حصوں یا ایونٹس کی ترتیب میں بھیجنے اور وصول کرنے کی اجازت دیتی ہے، بجائے اس کے کہ پورا جواب تیار ہونے کا انتظار کیا جائے۔ یہ خاص طور پر مفید ہے:

- بڑے فائلز یا ڈیٹا سیٹس کے لئے۔
- ریئل ٹائم اپڈیٹس (مثلاً، چیٹ، پروگریس بارز)۔
- طویل عرصے تک چلنے والے حسابات جہاں آپ صارف کو باخبر رکھنا چاہتے ہیں۔

یہاں وہ چیزیں ہیں جو آپ کو سٹریمنگ کے بارے میں اعلی سطح پر جاننا ضروری ہیں:

- ڈیٹا بتدریج فراہم کیا جاتا ہے، ایک بار میں نہیں۔
- کلائنٹ ڈیٹا کو جیسے ہی وہ آتا ہے پروسیس کر سکتا ہے۔
- محسوس شدہ تاخیر کو کم کرتا ہے اور صارف کے تجربے کو بہتر بناتا ہے۔

### سٹریمنگ کیوں استعمال کریں؟

سٹریمنگ استعمال کرنے کی وجوہات درج ذیل ہیں:

- صارفین فوری فیڈبیک حاصل کرتے ہیں، صرف آخر میں نہیں۔
- ریئل ٹائم ایپلیکیشنز اور جوابدہ UIs کو ممکن بناتا ہے۔
- نیٹ ورک اور کمپیوٹ وسائل کا زیادہ موثر استعمال۔

### سادہ مثال: HTTP سٹریمنگ سرور اور کلائنٹ

یہاں ایک سادہ مثال ہے کہ سٹریمنگ کو کیسے نافذ کیا جا سکتا ہے:

#### Python

**سرور (Python، FastAPI اور StreamingResponse استعمال کرتے ہوئے):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**کلائنٹ (Python، requests استعمال کرتے ہوئے):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

یہ مثال اس بات کی وضاحت کرتی ہے کہ سرور کلائنٹ کو پیغامات کی ایک سیریز بھیجتا ہے جیسے ہی وہ دستیاب ہوتے ہیں، بجائے اس کے کہ تمام پیغامات کے تیار ہونے کا انتظار کیا جائے۔

**یہ کیسے کام کرتا ہے:**

- سرور ہر پیغام کو جیسے ہی تیار ہو حاصل کرتا ہے۔
- کلائنٹ ہر حصہ کو وصول کرتا ہے اور جیسے ہی آتا ہے پرنٹ کرتا ہے۔

**ضروریات:**

- سرور کو سٹریمنگ ریسپانس استعمال کرنا چاہیے (مثلاً، FastAPI میں `StreamingResponse`)۔
- کلائنٹ کو ریسپانس کو سٹریم کے طور پر پروسیس کرنا چاہیے (`stream=True` requests میں)۔
- Content-Type عموماً `text/event-stream` یا `application/octet-stream` ہوتا ہے۔

#### Java

**سرور (Java، Spring Boot اور Server-Sent Events استعمال کرتے ہوئے):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**کلائنٹ (Java، Spring WebFlux WebClient استعمال کرتے ہوئے):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**Java نافذ کاری کے نکات:**

- سٹریمنگ کے لیے Spring Boot کے ریئیکٹیو اسٹیک میں `Flux` استعمال کرتا ہے
- `ServerSentEvent` ایونٹ ٹائپس کے ساتھ اسٹرکچرد ایونٹ سٹریمنگ فراہم کرتا ہے
- `WebClient` کے ساتھ `bodyToFlux()` ریئیکٹیو سٹریمنگ کی اجازت دیتا ہے
- `delayElements()` ایونٹس کے درمیان پراسیسنگ وقت کا اندازہ لگاتا ہے
- ایونٹس کے پاس بہتر کلائنٹ ہینڈلنگ کے لیے ٹائپس ہو سکتے ہیں (`info`, `result`)

### موازنہ: کلاسیکل سٹریمنگ بمقابلہ MCP سٹریمنگ

"کلاسیکل" انداز میں سٹریمنگ اور MCP میں سٹریمنگ کے طریقہ کار میں فرق کو اس طرح دکھایا جا سکتا ہے:

| خصوصیت                     | کلاسک HTTP سٹریمنگ             | MCP سٹریمنگ (نوٹیفیکیشنز)           |
|----------------------------|--------------------------------|----------------------------------------|
| مرکزی جواب                 | چنکس میں                      | سنگل، آخر میں                         |
| پروگریس اپڈیٹس            | ڈیٹا چنکس کے بطور بھیجے جاتے ہیں | نوٹیفیکیشنز کے طور پر بھیجے جاتے ہیں    |
| کلائنٹ کی ضروریات          | سٹریم کو پروسیس کرنا ضروری    | میسج ہینڈلر کو نافذ کرنا ضروری         |
| استعمال کا کیس             | بڑی فائلیں، AI ٹوکن سٹریمز    | پروگریس، لوگز، ریئل ٹائم فیڈبیک        |

### کلیدی فرق مشاہدہ کیے گئے

اس کے علاوہ، یہاں کچھ کلیدی فرق ہیں:

- **مواصلاتی پیٹرن:**  
  - کلاسک HTTP سٹریمنگ: سادہ چنکڈ ٹرانسفر انکوڈنگ استعمال کرتا ہے تاکہ ڈیٹا چنکس میں بھیجا جائے  
  - MCP سٹریمنگ: JSON-RPC پروٹوکول کے ساتھ ایک ساختہ نوٹیفیکیشن سسٹم استعمال کرتا ہے  

- **میسج فارمیٹ:**  
  - کلاسک HTTP: نئے لائنز کے ساتھ سادہ ٹیکسٹ چنکس  
  - MCP: میٹاڈیٹا کے ساتھ ساختہ LoggingMessageNotification آبجیکٹس  

- **کلائنٹ کی نافذ کاری:**  
  - کلاسک HTTP: سادہ کلائنٹ جو سٹریمنگ ریسپانسز کو پروسیس کرتا ہے  
  - MCP: زیادہ پیچیدہ کلائنٹ جو مختلف اقسام کے پیغامات کو پروسیس کرنے کے لئے میسج ہینڈلر رکھتا ہے  

- **پروگریس اپڈیٹس:**  
  - کلاسک HTTP: پروگریس مرکزی ریسپانس سٹریم کا حصہ ہوتی ہے  
  - MCP: پروگریس علیحدہ نوٹیفیکیشن میسجز کے ذریعے بھیجی جاتی ہے جبکہ مرکزی جواب آخر میں آتا ہے  

### سفارشات

کلاسیکل سٹریمنگ (جیسے کہ `/stream` اینڈپوائنٹ) کو نافذ کرنے اور MCP سٹریمنگ کے درمیان انتخاب کرتے وقت کچھ چیزیں ہم تجویز کرتے ہیں۔

- **سادہ سٹریمنگ ضروریات کے لیے:** کلاسیکل HTTP سٹریمنگ آسان ہے اور بنیادی سٹریمنگ ضروریات کے لیے کافی ہے۔

- **پیچیدہ، انٹرایکٹو ایپلیکیشنز کے لیے:** MCP سٹریمنگ ایک زیادہ ساختہ طریقہ فراہم کرتا ہے جس میں زیادہ میٹاڈیٹا اور نوٹیفیکیشنز اور حتمی نتائج کے درمیان تفریق ہوتی ہے۔

- **AI ایپلیکیشنز کے لیے:** MCP کا نوٹیفیکیشن سسٹم طویل مدتی AI کاموں کے لیے خاص طور پر مفید ہے جہاں آپ صارفین کو پروگریس کے بارے میں آگاہ رکھنا چاہتے ہیں۔

## MCP میں سٹریمنگ

تو، آپ نے اب تک کلاسیکل سٹریمنگ اور MCP سٹریمنگ کے درمیان کچھ سفارشات اور موازنہ دیکھا ہے۔ اب تفصیل میں چلتے ہیں کہ آپ MCP میں سٹریمنگ کو کس طرح فائدہ مند طریقے سے استعمال کر سکتے ہیں۔

MCP کے فریم ورک میں سٹریمنگ کے کام کرنے کے طریقہ کو سمجھنا ضروری ہے تاکہ آپ ریسپانسیو ایپلیکیشنز بنا سکیں جو طویل مدت کے کاموں کے دوران صارفین کو ریئل ٹائم فیڈبیک فراہم کریں۔

MCP میں، سٹریمنگ کا مطلب نہیں ہے کہ مرکزی جواب کو چنکس میں بھیجا جائے، بلکہ یہ ہے کہ کلائنٹ کو درخواست پروسیسنگ کے دوران **نوٹیفیکیشنز** بھیجی جائیں۔ یہ نوٹیفیکیشنز پروگریس اپڈیٹس، لوگز، یا دیگر ایونٹس پر مشتمل ہو سکتی ہیں۔

### یہ کیسے کام کرتا ہے

مرکزی نتیجہ اب بھی ایک ہی جواب کے طور پر بھیجا جاتا ہے۔ تاہم، نوٹیفیکیشنز الگ پیغامات کے طور پر پروسیسنگ کے دوران بھیجی جا سکتی ہیں اور اس طرح کلائنٹ کو ریئل ٹائم میں اپ ڈیٹ کیا جا سکتا ہے۔ کلائنٹ کو ان نوٹیفیکیشنز کو ہینڈل کرکے دکھانا آنا چاہیے۔

## نوٹیفیکیشن کیا ہے؟

ہم نے "نوٹیفیکیشن" کہا، MCP کے سیاق و سباق میں اس کا کیا مطلب ہے؟

نوٹیفیکیشن ایک پیغام ہے جو سرور سے کلائنٹ کو بھیجا جاتا ہے تاکہ طویل عرصے کے کام کے دوران پروگریس، حیثیت، یا دیگر ایونٹس کے بارے میں آگاہ کیا جا سکے۔ نوٹیفیکیشنز شفافیت اور صارف کے تجربے کو بڑھاتی ہیں۔

مثال کے طور پر، کلائنٹ کو نوٹیفیکیشن بھیجنی چاہیے جب سرور کے ساتھ ابتدائی ہینڈشیک مکمل ہو جائے۔

نوٹیفیکیشن JSON پیغام کی صورت میں کچھ اس طرح دکھتی ہے:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

نوٹیفیکیشنز MCP میں ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging) کے موضوع سے متعلق ہیں۔

> **پرانی ہوگئی اطلاع:** `2026-07-28` MCP اسپیسیفیکیشن ریلیز کینیڈیٹ لاگنگ پرمیٹو کو `stderr` (stdio ٹرانسپورٹس کے لیے) اور OpenTelemetry (ساختہ آبزرویبلیٹی کے لیے) کی ترجیح دیتے ہوئے منسوخ قرار دیتا ہے۔ لاگنگ `2025-11-25` میں کام کرتی رہے گی اور کسی بھی باقاعدہ منسوخی کے کم از کم ایک سال تک جاری رہے گی۔ مزید دیکھیں [MCP میں کیا تبدیلیاں ہو رہی ہیں: 2026-07-28 ریلیز کینیڈیٹ](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md)۔

لاگنگ کو کام کرنے کے لیے، سرور کو اسے فیچر/صلاحیت کے طور پر فعال کرنا ہوگا، کچھ اس طرح:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> استعمال ہونے والے SDK پر منحصر ہے، لاگنگ ڈیفالٹ طور پر فعال ہو سکتی ہے، یا آپ کو سرور کی کنفیگریشن میں واضح طور پر اسے فعال کرنا پڑ سکتا ہے۔

مختلف قسم کی نوٹیفیکیشنز ہیں:

| سطح       | وضاحت                             | استعمال کی مثال                   |
|-----------|----------------------------------|---------------------------------|
| debug     | تفصیلی ڈیبگنگ معلومات             | فنکشن کے داخلہ/خروج پوائنٹس     |
| info      | عمومی معلوماتی پیغامات             | آپریشن کی پیش رفت کی اپڈیٹس      |
| notice    | معمول کی اہم لیکن نمایاں ایونٹس     | کنفیگریشن میں تبدیلیاں           |
| warning   | انتباہی حالات                     | متروک فیچر کا استعمال            |
| error     | غلطی کے حالات                    | آپریشن کی ناکامیاں              |
| critical  | سنگین حالات                     | سسٹم کے جزو کی ناکامیاں          |
| alert     | فوری کارروائی ضروری ہے            | ڈیٹا کرپشن کا پتہ چلنا           |
| emergency | سسٹم ناقابل استعمال ہے            | مکمل سسٹم ناکامی                |

## MCP میں نوٹیفیکیشنز کو نافذ کرنا

MCP میں نوٹیفیکیشنز کو نافذ کرنے کے لیے، آپ کو دونوں سرور اور کلائنٹ سائیڈز کو ریئل ٹائم اپڈیٹس کو ہینڈل کرنے کے لیے ترتیب دینا ہوگا۔ یہ آپ کی ایپلیکیشن کو طویل مدتی آپریشنز کے دوران صارفین کو فوری فیڈبیک فراہم کرنے کی اجازت دیتا ہے۔

### سرور سائیڈ: نوٹیفیکیشن بھیجنا

آئیے سرور سائیڈ سے شروع کرتے ہیں۔ MCP میں، آپ ایسے ٹولز کی تعریف کرتے ہیں جو درخواستوں کی پروسیسنگ کے دوران نوٹیفیکیشنز بھیج سکتے ہیں۔ سرور کلائنٹ کو پیغامات بھیجنے کے لیے عموماً `ctx` کانٹیکسٹ آبجیکٹ استعمال کرتا ہے۔

#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

پچھلی مثال میں، `process_files` ٹول ہر فائل کی پروسیسنگ کے دوران تین نوٹیفیکیشنز کلائنٹ کو بھیجتا ہے۔ `ctx.info()` میتھڈ معلوماتی پیغامات بھیجنے کے لیے استعمال ہوتی ہے۔

مزید برآں، نوٹیفیکیشنز کو فعال کرنے کے لیے، یقینی بنائیں کہ آپ کا سرور سٹریمنگ ٹرانسپورٹ (جیسے `streamable-http`) استعمال کر رہا ہے اور آپ کا کلائنٹ نوٹیفیکیشنز کو پروسیس کرنے کے لیے میسج ہینڈلر نافذ کرتا ہے۔ یہاں یہ دکھایا گیا ہے کہ آپ سرور کو `streamable-http` ٹرانسپورٹ استعمال کرنے کے لیے کیسے ترتیب دیتے ہیں:

```python
mcp.run(transport="streamable-http")
```

#### .NET

```csharp
[Tool("A tool that sends progress notifications")]
public async Task<TextContent> ProcessFiles(string message, ToolContext ctx)
{
    await ctx.Info("Processing file 1/3...");
    await ctx.Info("Processing file 2/3...");
    await ctx.Info("Processing file 3/3...");
    return new TextContent
    {
        Type = "text",
        Text = $"Done: {message}"
    };
}
```

اس .NET مثال میں، `ProcessFiles` ٹول `Tool` اٹریبیوٹ کے ساتھ سجا ہوا ہے اور ہر فائل کی پروسیسنگ کے دوران تین نوٹیفیکیشنز کلائنٹ کو بھیجتا ہے۔ `ctx.Info()` طریقہ معلوماتی پیغامات بھیجنے کے لیے استعمال ہوتا ہے۔

.NET MCP سرور میں نوٹیفیکیشنز کو فعال کرنے کے لیے، یقینی بنائیں کہ آپ سٹریمنگ ٹرانسپورٹ استعمال کر رہے ہیں:

```csharp
var builder = McpBuilder.Create();
await builder
    .UseStreamableHttp() // Enable streamable HTTP transport
    .Build()
    .RunAsync();
```

### کلائنٹ سائیڈ: نوٹیفیکیشنز وصول کرنا

کلائنٹ کو نوٹیفیکیشنز کو وصول کرنے اور دکھانے کے لیے میسج ہینڈلر نافذ کرنا چاہیے۔

#### Python

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

پچھلے کوڈ میں، `message_handler` فنکشن چیک کرتا ہے کہ آیا آنے والا پیغام نوٹیفیکیشن ہے۔ اگر ہے تو، وہ نوٹیفیکیشن پرنٹ کرتا ہے؛ ورنہ اسے ایک معمولی سرور پیغام کے طور پر پروسیس کرتا ہے۔ نیز نوٹ کریں کہ `ClientSession` کو `message_handler` کے ساتھ شروع کیا گیا ہے تاکہ آنے والی نوٹیفیکیشنز کو ہینڈل کیا جا سکے۔

#### .NET

```csharp
// Define a message handler
void MessageHandler(IJsonRpcMessage message)
{
    if (message is ServerNotification notification)
    {
        Console.WriteLine($"NOTIFICATION: {notification}");
    }
    else
    {
        Console.WriteLine($"SERVER MESSAGE: {message}");
    }
}

// Create and use a client session with the message handler
var clientOptions = new ClientSessionOptions
{
    MessageHandler = MessageHandler,
    LoggingCallback = (level, message) => Console.WriteLine($"[{level}] {message}")
};

using var client = new ClientSession(readStream, writeStream, clientOptions);
await client.InitializeAsync();

// Now the client will process notifications through the MessageHandler
```

اس .NET مثال میں، `MessageHandler` فنکشن چیک کرتا ہے کہ آیا آنے والا پیغام نوٹیفیکیشن ہے۔ اگر ہے تو، وہ نوٹیفیکیشن پرنٹ کرتا ہے؛ ورنہ اسے ایک معمولی سرور پیغام کے طور پر پروسیس کرتا ہے۔ `ClientSession` کو میسج ہینڈلر کے ذریعے `ClientSessionOptions` میں شروع کیا گیا ہے۔

نوٹیفیکیشنز کو فعال کرنے کے لیے، یقینی بنائیں کہ آپ کا سرور سٹریمنگ ٹرانسپورٹ (جیسے `streamable-http`) استعمال کر رہا ہے اور آپ کا کلائنٹ نوٹیفیکیشنز کو پروسیس کرنے کے لیے میسج ہینڈلر نافذ کرتا ہے۔

## پروگریس نوٹیفیکیشنز اور منظرنامے

یہ سیکشن MCP میں پروگریس نوٹیفیکیشنز کا تصور، ان کی اہمیت، اور Streamable HTTP کا استعمال کرتے ہوئے انہیں کیسے نافذ کیا جائے سمجھاتا ہے۔ آپ کو اپنی سمجھ کو مضبوط کرنے کے لیے ایک عملی مشق بھی ملے گی۔

پروگریس نوٹیفیکیشنز ریئل ٹائم پیغامات ہوتے ہیں جو سرور سے کلائنٹ کو طویل مدتی کاموں کے دوران بھیجے جاتے ہیں۔ پورے عمل کے ختم ہونے کا انتظار کرنے کے بجائے، سرور کلائنٹ کو موجودہ حیثیت کے بارے میں اپ ڈیٹ کرتا رہتا ہے۔ یہ شفافیت، صارف کا تجربہ بڑھاتا ہے، اور ڈیبگنگ آسان بناتا ہے۔

**مثال:**

```text

"Processing document 1/10"
"Processing document 2/10"
...
"Processing complete!"

```

### پروگریس نوٹیفیکیشنز کیوں استعمال کریں؟

پروگریس نوٹیفیکیشنز کئی وجوہات کی بنا پر ضروری ہیں:

- **بہتر صارف تجربہ:** صارفین کام کی پیش رفت کو اپڈیٹس کی صورت میں دیکھتے ہیں، صرف آخر میں نہیں۔  
- **ریئل ٹائم فیڈبیک:** کلائنٹس پروگریس بارز یا لوگز دکھا سکتے ہیں، جس سے ایپ متحرک محسوس ہوتی ہے۔  
- **آسان ڈیبگنگ اور مانیٹرنگ:** ڈویلپرز اور صارفین دیکھ سکتے ہیں کہ عمل کہاں سست یا رکا ہوا ہے۔  

### پروگریس نوٹیفیکیشنز کیسے نافذ کریں

یہاں یہ بتایا گیا ہے کہ MCP میں پروگریس نوٹیفیکیشنز کیسے نافذ کرنی ہیں:

- **سرور پر:** جب ہر آئٹم پروسیس ہو رہا ہو، `ctx.info()` یا `ctx.log()` استعمال کریں تاکہ نوٹیفیکیشنز بھیجی جا سکیں۔ یہ مرکزی نتیجہ تیار ہونے سے پہلے کلائنٹ کو پیغام بھیجتا ہے۔  
- **کلائنٹ پر:** ایک میسج ہینڈلر نافذ کریں جو نوٹیفیکیشنز کو سنتا اور دکھاتا ہے جیسے ہی وہ آتی ہیں۔ یہ ہینڈلر نوٹیفیکیشنز اور حتمی نتیجہ کے درمیان تمیز کرتا ہے۔  

**سرور کی مثال:**


#### پائتھن

```python

async def process_files(message: str, ctx: Context) -> TextContent:
    for i in range(1, 11):
        await ctx.info(f"Processing document {i}/10")
    await ctx.info("Processing complete!")
    return TextContent(type="text", text=f"Done: {message}")
```

**کلائنٹ کی مثال:**

#### پائتھن

## سیکیورٹی کے تحفظات

جب MCP سرورز کو HTTP-بیسڈ ٹرانسپورٹس کے ساتھ نافذ کیا جاتا ہے، تو سیکیورٹی ایک نہایت اہم مسئلہ بن جاتی ہے جس کے لیے متعدد حملے کے راستوں اور حفاظتی طریقوں پر غور و خوض کرنا ضروری ہوتا ہے۔

### جائزہ

MCP سرورز کو HTTP کے ذریعے ظاہر کرتے وقت سیکیورٹی بہت اہم ہوتی ہے۔ اسٹریم ایبل HTTP نئے حملے کے راستے متعارف کرواتا ہے اور محتاط کنفیگریشن کا تقاضا کرتا ہے۔

### اہم نکات

- **Origin ہیڈر کی تصدیق**: DNS ری بائنڈنگ حملوں سے بچنے کے لیے ہمیشہ `Origin` ہیڈر کی تصدیق کریں۔
- **لوکل ہوسٹ بائنڈنگ**: مقامی ترقی کے لیے سرورز کو `localhost` سے باندھیں تاکہ انہیں عوامی انٹرنیٹ پر ظاہر نہ کیا جائے۔
- **تصدیق کاری**: پروڈکشن تنصیبات کے لیے تصدیق کاری نافذ کریں (مثلاً API کیز، OAuth)۔
- **CORS**: Cross-Origin Resource Sharing (CORS) پالیسیز کو کنفیگر کریں تاکہ رسائی محدود ہو۔
- **HTTPS**: ٹریفک کو انکرپٹ کرنے کے لیے پروڈکشن میں HTTPS استعمال کریں۔

### بہترین طریقے

- آنے والی درخواستوں پر بغیر تصدیق کے کبھی بھروسہ نہ کریں۔
- تمام رسائی اور غلطیوں کو لاگ کریں اور نگرانی کریں۔
- سیکیورٹی کے مسائل کو حل کرنے کے لیے وقتاً فوقتاً انحصار کو اپڈیٹ کریں۔

### چیلنجز

- سیکیورٹی اور ترقی کی آسانی کے درمیان توازن قائم کرنا
- مختلف کلائنٹ کے ماحول کے ساتھ مطابقت کو یقینی بنانا

## SSE سے Streamable HTTP پر اپ گریڈ کرنا

جو ایپلیکیشنز اس وقت Server-Sent Events (SSE) استعمال کر رہی ہیں، ان کے لیے Streamable HTTP کی طرف منتقلی بہتر صلاحیتیں فراہم کرتی ہے اور MCP کی طویل مدتی پائیداری بڑھاتی ہے۔

### کیوں اپ گریڈ کریں؟

SSE سے Streamable HTTP پر اپ گریڈ کرنے کی دو اہم وجوہات ہیں:

- Streamable HTTP بہتر اسکیلیبلیٹی، مطابقت، اور زیادہ جامع نوٹیفیکیشن سپورٹ فراہم کرتا ہے بہ نسبت SSE کے۔
- یہ نئے MCP ایپلیکیشنز کے لیے تجویز کردہ ٹرانسپورٹ ہے۔

### منتقلی کے مراحل

اپنے MCP ایپلیکیشنز میں SSE سے Streamable HTTP پر منتقلی کا طریقہ درج ذیل ہے:

- **سرور کوڈ اپ ڈیٹ کریں** تاکہ `mcp.run()` میں `transport="streamable-http"` استعمال ہو۔
- **کلائنٹ کوڈ اپ ڈیٹ کریں** تاکہ SSE کلائنٹ کی جگہ `streamablehttp_client` استعمال ہو۔
- **کلائنٹ میں پیغام ہینڈلر نافذ کریں** تاکہ نوٹیفیکیشنز کو پروسیس کیا جا سکے۔
- **موجودہ آلات اور ورک فلو کے ساتھ مطابقت کی جانچ کریں**۔

### مطابقت برقرار رکھنا

سفارش کی جاتی ہے کہ منتقلی کے دوران موجودہ SSE کلائنٹس کے ساتھ مطابقت کو برقرار رکھا جائے۔ یہاں چند حکمت عملیاں ہیں:

- آپ دونوں SSE اور Streamable HTTP کو مختلف اینڈپوائنٹس پر چلا کر سپورٹ کر سکتے ہیں۔
- کلائنٹس کو نئی ٹرانسپورٹ کی جانب تدریجی منتقلی کرائیں۔

### چیلنجز

منتقلی کے دوران درج ذیل چیلنجز پر توجہ دیں:

- اس بات کو یقینی بنانا کہ تمام کلائنٹس اپڈیٹ ہو جائیں
- نوٹیفیکیشن کی فراہمی میں فرق کو سنبھالنا

## سیکیورٹی کے تحفظات

جب بھی کسی سرور کا نفاذ کیا جائے، خصوصاً MCP میں HTTP-بیسڈ ٹرانسپورٹس جیسے Streamable HTTP کا استعمال کرتے ہوئے، سیکیورٹی کو اولین ترجیح دی جانی چاہیے۔

جب MCP سرورز کو HTTP-بیسڈ ٹرانسپورٹس کے ساتھ نافذ کیا جاتا ہے، تو سیکیورٹی ایک نہایت اہم مسئلہ بن جاتی ہے جس کے لیے متعدد حملے کے راستوں اور حفاظتی طریقوں پر غور و خوض کرنا ضروری ہوتا ہے۔

### جائزہ

MCP سرورز کو HTTP کے ذریعے ظاہر کرتے وقت سیکیورٹی بہت اہم ہوتی ہے۔ اسٹریم ایبل HTTP نئے حملے کے راستے متعارف کرواتا ہے اور محتاط کنفیگریشن کا تقاضا کرتا ہے۔

یہاں کچھ اہم سیکیورٹی تحفظات ہیں:

- **Origin ہیڈر کی تصدیق**: DNS ری بائنڈنگ حملوں سے بچنے کے لیے ہمیشہ `Origin` ہیڈر کی تصدیق کریں۔
- **لوکل ہوسٹ بائنڈنگ**: مقامی ترقی کے لیے سرورز کو `localhost` سے باندھیں تاکہ انہیں عوامی انٹرنیٹ پر ظاہر نہ کیا جائے۔
- **تصدیق کاری**: پروڈکشن تنصیبات کے لیے تصدیق کاری نافذ کریں (مثلاً API کیز، OAuth)۔
- **CORS**: Cross-Origin Resource Sharing (CORS) پالیسیز کو کنفیگر کریں تاکہ رسائی محدود ہو۔
- **HTTPS**: ٹریفک کو انکرپٹ کرنے کے لیے پروڈکشن میں HTTPS استعمال کریں۔

### بہترین طریقے

مزید برآں، MCP اسٹریمینگ سرور میں سیکیورٹی نافذ کرتے وقت درج ذیل بہترین طریقے اپنائیں:

- آنے والی درخواستوں پر بغیر تصدیق کے کبھی بھروسہ نہ کریں۔
- تمام رسائی اور غلطیوں کو لاگ کریں اور نگرانی کریں۔
- سیکیورٹی کے مسائل کو حل کرنے کے لیے وقتاً فوقتاً انحصار کو اپڈیٹ کریں۔

### چیلنجز

MCP اسٹریمینگ سرورز میں سیکیورٹی نافذ کرتے وقت چند چیلنجز کا سامنا ہوگا:

- سیکیورٹی اور ترقی کی آسانی کے درمیان توازن قائم کرنا
- مختلف کلائنٹ کے ماحول کے ساتھ مطابقت کو یقینی بنانا

### مشق: اپنا خود کا اسٹریمینگ MCP ایپ بنائیں

**منظر نامہ:**
ایک MCP سرور اور کلائنٹ بنائیں جہاں سرور اشیاء کی فہرست (مثلاً فائلیں یا دستاویزات) کو پروسیس کرے اور ہر پروسیس کی گئی شے کے لیے نوٹیفیکیشن بھیجے۔ کلائنٹ کو چاہیے کہ وہ ہر نوٹیفیکیشن کو پہنچتے ہی دکھائے۔

**مراحل:**

1. ایک سرور ٹول نافذ کریں جو فہرست کو پروسیس کرے اور ہر شے کے لیے نوٹیفیکیشن بھیجے۔
2. ایک کلائنٹ نافذ کریں جس میں پیغام ہینڈلر ہو تاکہ نوٹیفیکیشنز حقیقی وقت میں دکھائے جا سکیں۔
3. اپنے نفاذ کا جائزہ لینے کے لیے سرور اور کلائنٹ دونوں کو چلائیں اور نوٹیفیکیشنز کا مشاہدہ کریں۔
